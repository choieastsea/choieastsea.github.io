오늘은 파이썬에서 종속성 라이브러리를 관리하는 패키지 매니저인 poetry에 대해 알아보겠습니다.


## 종속성 관리의 필요성

python은 혼자(외부 라이브러리) 코딩하기 참 편하지만, 다수와 같이 작업하는 순간 고려해야할 점들이 굉장히 많아집니다. 

가장 큰 것이 **패키지 종속성으로 인한 문제**인데요. 시스템에 설치된 파이썬 라이브러리와 호환되지 않거나, 설치해야할 라이브러리들 간의 버전이 충돌되는 문제가 발생할 수 있습니다.

이에 두가지 방법을 사용합니다.

1. 가상환경
	프로젝트마다 독립적인 python 환경을 갖도록 합니다. global package는 지양하고, 프로젝트마다 고유한 `virtual environment`를 만들어 사용하도록 합니다.
	
	python3에서는 `python -m venv {가상환경 이름}`의 꼴로 현재 디렉토리에 가상환경에 대한 폴더가 생기고, `source` 등의 명령어를 통해 가상환경을 활성화할 수 있습니다. [참고](https://docs.python.org/3/library/venv.html#how-venvs-work)
	
2. 종속성 파일
	라이브러리간 의존성과 그로 인한 충돌을 관리하기 위해서는 라이브러리의 버전을 관리해야 합니다. 보통 `requirements.txt` 파일에 의존하는 패키지의 버전을 명시하도록 하고, 사용자는 `pip install -r requirements.txt`로 이를 설치하는 것이 관례입니다.


> \[  python 라이브러리 충돌 문제 해결 방법 (basic) \]
> 
> 1. python 자체적으로, `venv` 와 `pip install`을 통하여 프로젝트 라이브러리를 독립적으로 관리할 수 있습니다.
> 
> 2. 또한, 관례적으로 `requirements.txt`에 최소한의 라이브러리 버전 정보를 명시합니다.
{:.prompt-tip }
## poetry의 등장

물론, `venv` 모듈과 `pip + requirements.txt` 의 조합으로 naive하게 문제를 해결할 수 있지만, requirements.txt는 문제가 있습니다.
1. 라이브러리 간의 충돌
	라이브러리의 의존성 패키지끼리 충돌할 경우, 이를 추적하고 requirements.txt를 **일일이 수정**해야 합니다.
2. requirements.txt를 수정하는 것의 문제
	javascript의 패키지 관리자인 `yarn`, `npm`은 별도의 **lock file**을 두어 버전의 필수 조건 뿐 아니라 실제 설치된 버전을 명시합니다. 또한 해당 lock file를 개발자가 임의로 수정하지 못하도록 하여, lock file이 있다면 일치된 라이브러리 환경을 갖을 수 있게 됩니다.

	 `requirements.txt`의 경우, `pip freeze > requirements.txt`와 같이 freeze 할 수는 있지만, 위의 충돌을 해결하는 과정에서 개발자가 직접 수정해야하는데 이 과정으로 해결될거라 보장할 수 없습니다.

	또한, conventional하게 *의존성 패키지 목록을 txt로 제공하는 것도 문제가 될 수 있겠습니다*.

앞서 언급했던 문제들을 해결하기 위해 `poetry`가 등장합니다.

**추가로 프로젝트를 build하고 배포하는 기능까지도 지원하여, 하나로 기존의 많은 모듈들을 대체**하게 되었습니다. 

## poetry 주요 명령어 정리

poetry는 시스템에 글로벌하게 설치하는 것이 좋습니다. [참고](https://python-poetry.org/docs/#installation)

1. 초기 설정 - `poetry new` / `poetry init`
	
    `poetry`를 설치한 이후, `poetry init`으로 interactive하게 *기존 디렉토리의 package manager를 poetry로 설정*할 수 있습니다. 기존 프로젝트가 이미 있다면 poetry init을 수행하면 됩니다.
    
    아니면 `poetry new` 명령어로 프로젝트 시작을 수행할 수도 있습니다.
    
	```shell
	> poetry new hello-project
	Created package hello_project in hello-project
	
	> ls hello-project
	README.md  hello_project  pyproject.toml  tests
	```
	모든 입력을 완료하면 `pyproject.toml`이 생깁니다. 
	
	new 명령어로 생성한 경우에는 **프로젝트 폴더와 README, 그리고 테스트 폴더**까지 conventional하게 생성해줍니다!

	```toml
	[tool.poetry]
	name = "hello-project"
	version = "0.1.0"
	description = ""
	authors = ["choieastsea <email>"]
	readme = "README.md"

	[tool.poetry.dependencies]
	python = "^3.9"
	  
	[build-system]
	requires = ["poetry-core"]
	build-backend = "poetry.core.masonry.api"
	```
	`toml` 파일은 `[ ]` 안에 섹션 이름, 그리고 key = value의 꼴로 데이터를 표현하는 방법입니다. [공식 문서](https://toml.io/en/)
	`pyproject.toml` 에는 프로젝트의 정보를 담고 있습니다.

	- 프로젝트 기본 정보
	- 사용하는 라이브러리
	- 빌드 정보

2. 가상환경 활성화 - `poetry shell`

	poetry 가상환경은 프로젝트 루트 디렉토리(pyproject.toml이 있는 곳)에서 `poetry shell` 명령어를 통해 활성화 할 수 있습니다.

	하지만 기본적으로 poetry 설치 위치에 가상환경이 세팅되는데 이는 독립적이긴 하지만, 라이브러리의 소스 코드를 추적하고 싶을 때 곤란합니다.
	
	`poetry config virtualenvs.in-project true` 명령어를 통하여 이후 생성할 가상환경을 현재 디렉토리에 생기게 합니다. (혹은 poetry.toml에 in-project 키를 true로 정해줘도 됩니다)

	> `pyproject.toml`은 project에 대한 정보가 담기고,
	>
	> `poetry.toml`은 poetry에 대한 설정 정보가 담기므로 구분합시다!
	{:.prompt-tip }
	

	`poetry shell`로 가상환경을 생성하고, (이미 생성되었다면) 활성화할 수 있습니다. 가상환경이 활성화되면 user name 옆에 `({가상환경 이름})`의 꼴로 나타나게 됩니다.
	
3. 라이브러리 추가 - `poetry add` / `poetry remove`
	
	이제, 라이브러리를 추가해봅시다! toml 파일에 직접 작성할 수도 있지만, 보통 `poetry add`로 자동으로 추가되도록 합니다.
	
	```shell
	> poetry add aiohttp
	Using version ^3.9.3 for aiohttp

	Updating dependencies
	Resolving dependencies... (0.3s)

	Package operations: 8 installs, 0 updates, 0 removals

	  • Installing frozenlist (1.4.1)
	  • Installing idna (3.6)
	  • Installing multidict (6.0.5)
	  • Installing aiosignal (1.3.1)
	  • Installing async-timeout (4.0.3)
	  • Installing attrs (23.2.0)
	  • Installing yarl (1.9.4)
	  • Installing aiohttp (3.9.3)
	```

	위와 같이 의존하고 있는 프로젝트를 먼저 설치한 이후에 요청한 라이브러리를 설치하는 것을 확인할 수 있습니다.

4. poetry.lock
	
	라이브러리를 설치하면 `poetry.lock` 파일이 생깁니다. 해당 파일은 사용자가 임의로 라이브러리의 종속관계를 정의하지 못하도록 hash값을 추가합니다.
	
	파일 시작부터 `# This file is automatically @generated by Poetry 1.7.1 and should not be changed by hand.`라는 후덜덜한 멘트가 써있으니 건들지 맙시다.

	poetry 명령어를 통해 패키지를 삭제하거나 버전을 변경하면 해당 사항이 바로 반영됩니다.
	
	> Q. poetry.lock 파일을 커밋해야할까요?
	> 
	> 협업시에는 poetry.lock을 일치시켜 같은 종속성 환경을 유지시키는 것이 좋습니다! 라이브러리가 복잡할수록 lock file을 같이 배포하는 것이 좋습니다.
	{:.prompt-tip }
	
5. cli 실행 - `poetry run`
	
	가상환경에서 설치한 라이브러리를 cli 환경으로 실행하고 싶다면 `poetry run {library}`와 같이 실행할 수 있습니다. python 실행도 `poetry run python main.py`와 같이 실행할 수 있습니다.
